# P0/P1/P2 execution ledger

Status: active implementation ledger, reviewed 2026-07-15.

This file is the completion contract for the request to implement every P0, P1 and P2 scanner
epic. A checkbox may be marked complete only when the implementation is connected to its intended
runtime surface and the named verification evidence exists. A pure engine is valuable progress but
does not prove a continuous scanner, browser workflow or account-aware integration.

The evidence labels in this ledger are deliberately independent:

| Evidence layer | What it proves | What it does not prove |
| --- | --- | --- |
| Engine | Pure bounded calculation/protocol code exists | Server mounting, a running feed or browser access |
| Runtime | The server can mount/start the component when configured | That this deployment activated it or received live data |
| Browser | A localized EN/RU/KK workflow/diagnostic is connected | Venue correctness, private entitlement or fillability |
| Deterministic | Fixtures, unit/contract/failure tests pass | Current public connectivity or sustained operation |
| Public canary | Credential-free venue data was observed at the recorded time/network | Private account evidence, soak or production readiness |
| Private evidence | Authenticated account/order/fill state was observed in its explicit scope | Other venues/products or mainnet readiness |
| Production-ready | All required runtime, sustained, operational, legal and private gates pass | Never inferred from any earlier layer |

Some rows have delivered slices at several layers, but the thirty-row P0/P1/P2 ledger is **not
complete**. The remaining-proof column is normative.

The previously excluded funded 7–14 day Binance/Bybit private testnet soak is not part of this
ledger. Nothing here changes the experimental live-trading label or authorizes mainnet execution.

## Completion gates

The full ledger is complete only when all thirty rows have:

1. a runtime-connected implementation rather than documentation-only or caller-supplied scaffolding;
2. deterministic unit, contract and failure-injection coverage;
3. bounded resource/backpressure behavior;
4. EN/RU/KK user-facing wording where a browser surface exists;
5. generated API/SDK contracts where a public surface exists;
6. production-build Chromium, Firefox and WebKit coverage for browser workflows;
7. explicit research/private/mainnet capability labels with no inferred authorization.

The current repository does not yet satisfy every gate for every row; in particular deterministic
engine/browser delivery, a one-time public canary and an implemented operator configuration file
must not be collapsed into a production-ready claim.

Verification commands and evidence counts are recorded in [Arbitrage test matrix](ARBITRAGE_TEST_MATRIX.md)
and [Testing strategy](TESTING_STRATEGY.md).
