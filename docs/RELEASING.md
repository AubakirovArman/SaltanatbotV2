# Release policy and artifact verification

SaltanatbotV2 publishes the same built application shape through four maturity channels. A channel describes stability, not trading profitability or production readiness.

| Channel | Version/tag | Publication | Retention/intent |
| --- | --- | --- | --- |
| Nightly | `nightly-YYYYMMDD-<sha>` | Scheduled or manually dispatched Actions artifact | 14-day development snapshot |
| Alpha | `vMAJOR.MINOR.PATCH-alpha.N` | Tagged prerelease | Early integration testing; breaking changes expected |
| Beta | `vMAJOR.MINOR.PATCH-beta.N` or `-rc.N` | Tagged prerelease | Feature-frozen validation candidate |
| Stable | `vMAJOR.MINOR.PATCH` | Tagged release | Passed documented release gates; live trading remains operator-controlled |

Only a pushed, supported Git tag creates a GitHub Release. Manual runs build attestable artifacts without inventing or moving a Git tag.

## Produced assets

Each run produces:

- a reproducible `saltanatbotv2-<version>.tar.gz` application archive;
- `release-info.json` with channel, commit and `SOURCE_DATE_EPOCH`;
- an internal and external SHA-256 distribution manifest covering every extracted file;
- an SPDX JSON software bill of materials generated by Syft;
- a machine-readable controlled-corruption/verified-rollback drill report;
- `SHA256SUMS` for the archive, metadata, distribution manifest, drill evidence and SBOM;
- Sigstore bundles for SLSA build provenance and the SBOM attestation.

The archive contains built backend/frontend output, workspace runtime packages, lockfile, deployment files and documentation. It never contains `backend/data`, `.env`, exchange credentials, local databases or `node_modules`.

## Local packaging check

```bash
npm ci
npm run check
npm test
npm run build
ALLOW_DIRTY_RELEASE=1 npm run release:package -- --channel nightly --version nightly-local-$(git rev-parse --short HEAD)
npm run release:rollback-drill -- --distribution .release-staging/saltanatbotv2-nightly-local-$(git rev-parse --short HEAD) --output release/rollback-drill.json
```

The packager rejects a dirty tree by default; `ALLOW_DIRTY_RELEASE=1` is only for explicitly local, non-publishable experiments and is recorded in metadata. Local packaging creates the deterministic archive and metadata. SBOM generation and keyless signing run in the protected GitHub workflow because the signature identity is bound to the repository and workflow through short-lived OIDC credentials.

## Verify a downloaded release

From the directory containing the assets:

```bash
sha256sum --check SHA256SUMS
gh attestation verify saltanatbotv2-v0.2.0.tar.gz --repo OWNER/REPOSITORY
gh attestation verify saltanatbotv2-v0.2.0.tar.gz --repo OWNER/REPOSITORY --predicate-type https://spdx.dev/Document/v2.3
```

Verification must confirm the expected repository, workflow identity and digest. A valid attestation proves where an artifact was built and that it has not changed; it does not prove the software is defect-free or that a strategy is safe.

## Maintainer release sequence

1. Ensure CI, browser tests and protected exchange testnet smoke are green for the selected commit.
2. Review `CHANGELOG.md`, migrations, Pine compatibility changes and security notices.
3. Create an annotated, signed tag matching the channel and push it without moving an existing release tag.
4. Let `.github/workflows/release.yml` build from the tag; never upload a separately built replacement archive.
5. Require the distribution rollback drill to pass and retain its attested JSON evidence.
6. Verify checksums and attestations from a clean machine before announcing the release.
7. Enable GitHub immutable releases where the repository plan supports them.
