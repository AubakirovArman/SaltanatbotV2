# Сұраныс бойынша техникалық скринер

Ағылшынша түпнұсқа: [SCREENER.md](../SCREENER.md).

Орысша нұсқа: [ru/SCREENER.md](../ru/SCREENER.md).

Бұл құжат PostgreSQL schema 14 енгізген R5.2.1 техникалық скринерін
сипаттайды. Бұл — Binance spot USDT әлемі бойынша сұраныс бойынша іске
қосылатын, тек зерттеуге арналған индикаторлық скан. Әр payload
`researchOnly: true` және `executionPermission: false` мәндерін алып жүреді.
Скринер ордер орналастыра алмайды, актив қарызға ала алмайды, маржаны өзгерте
алмайды, биржа сұрауына қол қоя алмайды және сауда рөлін бере алмайды.

## R5.2.1 ауқымы

Бір скан пішіні қолданылады:

- Binance-тің ашық spot нарықтары бойынша kind `technical`, тек last price;
- `5m`, `15m`, `1h`, `4h` және `1d` таймфреймдері;
- каталогтың өсу ретімен сұрыпталған 10–200 USDT символынан тұратын әлем;
- тек жабылған candle-дар бойынша бағаланатын жеті түрдің 1–12 фильтрі;
- детерминистік сұрыптау, ең көбі 100 нәтиже жолы;
- ревизиялары мен архиві бар owner-scoped серверлік пресеттер;
- бар compute-job retention арқылы шектелген run тарихы;
- символ, таймфрейм және индикатор контексін сақтайтын графикке өту.

R5.2.1 құрамына кірмейді: жоспарланған немесе үздіксіз қайталанатын скандар,
Bybit әлемі және USDT емес квота активтері. Сканды серверлік алертке
айналдыру да бұған кірмеген; ол бөлек R5.3a инкрементімен rule kind
`screener` ретінде жеткізіледі — қараңыз:
[Скринер алерттері (R5.3a)](./ALERTS.md). Бұл инкрементте
анықтама контракты тек `exchange: "binance"`, `marketType: "spot"` және
`priceType: "last"` қабылдайды; schema кейінгі биржаларға нұсқаны бұзбай орын
қалдырады.

Скринер биржа credential-дарын ешқашан оқымайды. Ол бір ашық 24 сағаттық
ticker batch пен қатаң ашық жабылған candle-дарды қолданады және synthetic,
cached, private немесе қол қойылмаған алмастыру evidence-ін қабылдамайды.

## HTTPS жоқ кездегі HTTP deployment шекарасы

Қазіргі pre-HTTPS релиз TLS-ті әдейі баптамайды. Сондықтан login password пен
session cookie тек сенімді жергілікті желі, private VPN немесе SSH tunnel
арқылы берілуі керек. Бұл build-ті жалпыға ашық интернет login service ретінде
жарияламаңыз және оған биржаның private key-лерін қоспаңыз. HTTPS — бөлек
release gate.

## Деректер ағыны

```text
аутентификацияланған браузер
  -> POST /api/jobs (kind screener, идемпотентті clientRequestId)
  -> иесіне бекітілген тұрақты compute job
  -> research worker lease
  -> бір ашық 24h ticker batch + әр символға қатаң жабылған candle-дар
  -> таза детерминистік фильтр бағалауы
  -> шектелген сериализацияланған screener-run-result-v1
  -> GET /api/jobs/:id polling
  -> нәтижелер кестесі / графикке өту
```

Run — кәдімгі compute job. Браузер оны идемпотентті `clientRequestId` (8–128
таңба) арқылы кезекке қояды, job-ты әр 2 секунд сайын сұрайды, клиенттік
дедлайны 120 секунд, ал иесі run-нан бас тартса best-effort cancel сұрайды.
Бірдей күтіп тұрған payload-тарды jobs ішкі жүйесі дедупликациялайды; бір иеде
ең көбі бес белсенді және бір орындалып жатқан job болады.

## Анықтама сипаттамасы

`screener-definition-v1` құжаты дәл кілттермен талданады; белгісіз өрістер
қабылданбайды. Ол мыналарды қамтиды:

| Өріс | Мәні |
| --- | --- |
| `name` | 1–120 таңба |
| `exchange`, `marketType`, `priceType` | тек `binance`, `spot`, `last` |
| `timeframe` | `5m`, `15m`, `1h`, `4h` немесе `1d` |
| `universeLimit` | 10–200 бүтін саны |
| `sort` | кілт: `quoteVolume24h`, `change24hPercent`, `lastClose`, `symbol`, `rsi` немесе `atrPercent`; бағыт `asc`/`desc` |
| `filters` | 1–12 фильтр объектісі |
| `researchOnly`, `executionPermission` | міндетті түрде `true` / `false` |

Сандық шектер — алерт контракттарындағыдай ондық жолдар; олар ешқашан үнсіз
дөңгелектелмейді.

## Фильтр сипаттамасы

| Kind | Параметрлер | Сәйкестік |
| --- | --- | --- |
| `price` | `min?`, `max?` (кемінде бір шек) | соңғы close шектер ішінде |
| `quote-volume-24h` | `min` (USDT) | 24h quote volume ≥ `min` |
| `change-24h-percent` | `min?`, `max?` (−100…10000, кемінде бірі) | 24h өзгеріс шектер ішінде |
| `rsi` | `period` 2–200, `condition` `above`/`below`, `value` 0–100 | close бойынша RSI шартты қанағаттандырады |
| `ma-cross` | `fastType`/`slowType` `ema`/`sma`, `fastPeriod` < `slowPeriod`, `state` | `fast-above`, `fast-below`, `crossed-up` немесе `crossed-down` |
| `macd` | `fast` < `slow`, `signal` | `histogram-above-zero`, `histogram-below-zero`, `crossed-up`, `crossed-down` |
| `atr-percent` | `period` 2–200, `condition`, `value` 0–1000 | соңғы close-тың пайызындағы ATR шартты қанағаттандырады |

Соңғы жабылған candle — бағалау candle-ы. Cross күйлері дәл соңғы екі жабылған
candle-ды салыстырады: `crossed-up` деген fast мәні алдыңғы жабылған candle-да
жоғары болмағанын және бағалау candle-ында жоғары екенін білдіреді. Индикатор
математикасы `@saltanatbotv2/strategy-core` пакетінен алынады (RSI, SMA, EMA,
MACD, ATR) — график қолданатын пакеттер тобы; графикпен паритетті дәлелдейтін
де осы.

## Жабылған candle-дар және fail-closed қолжетімсіздік

Скринер тек финалдық candle-дарды бағалайды. Қалыптасып жатқан ұш алынып
тасталады, ал қалған терезе алерт evaluator-ы қолданатын жабылған candle
валидациясынан өтуге тиіс: жоқ, қалыптасып жатқан, болашақ, ескірген, үзілген,
өлшемнен асқан немесе бұзылған терезелер fail closed береді.

Деректері толық емес символ **нөл емес, қолжетімсіз** болады. Жылыну (warm-up)
терезесіндегі индикатор, жоқ ticker жолы немесе таусылған run бюджеті символды
фильтрге жалған `0` қоюдың орнына нақты себеппен бағалаудан шығарады. Нәтижеде
`universe.requested/evaluated/matched/unavailable` санауыштары және дәл себеп
жолдары бар `unavailableReasons` картасы болады, соның ішінде:

| Себеп | Мәні |
| --- | --- |
| `ticker-unavailable` | символға жарамды ашық 24h ticker жолы жоқ |
| `missing-candles` | провайдер жабылған candle қайтарған жоқ |
| `indicator-warm-up` | сұралған индикаторға жабылған candle жеткіліксіз |
| `run-budget-exhausted` | 90 секундтық бюджет осы символға дейін таусылды |
| `upstream-unavailable` | шектелген ашық оқу сәтсіз аяқталды |
| `row-out-of-range` | сәйкес мән шектелген контрактта көрсетілмейді |

Егер 24 сағаттық ticker batch-тың өзі талданбаса, бүкіл run жартылай баға
деректерімен жалғаспай, fail closed аяқталады.

## Детерминизм және нәтиже пішіні

Нәтиже — анықтама хэші, генерация уақыты, жабылған candle уақыт ауқымы және ең
көбі 100 жолы бар `screener-run-result-v1` құжаты. Жолдар анықтамадағы сұрыптау
кілтімен сұрыпталады, undefined мәндер соңында, тең болғанда символ өсу
ретімен, сондықтан бірдей әлем бірдей candle-дармен бірдей нәтиже береді. 100
символдан көп сәйкес келсе, нәтиже деректерді үнсіз тастамай,
`rowsTruncated: true` қояды. Ондық метрикалар `toFixed(8)` пішімімен, соңғы
нөлдері қырқылып жазылады.

## Иелік және авторизация

Пресетті әр оқу мен жазу иесін аутентификацияланған database сессиясынан
шығарады және қосымша сол user ID-мен `X-SBV2-Expected-User` тақырыбын талап
етеді. Мутацияларға кәдімгі CSRF тақырыбы қажет. Репозиторий транзакциялары
пайдаланушының белсенділігін, `must_change_password = false`, ағымдағы
authorization revision-ды және актор мен иенің сәйкестігін қайта тексереді —
дәл алерт репозиторийіндегідей. Әкімшілерге басқа иенің пресеттерін оқу немесе
өзгерту жолы берілмейді. Ашық проекцияларда `owner_user_id`,
`authorization_revision` және `definition_hash` ешқашан болмайды.

Пресет жасау `clientId` бойынша идемпотентті: сол анықтаманы қайталау бар
пресетті қайтарады, ал сол `clientId` астындағы басқа анықтама — `409`
қақтығысы. Жаңарту мен архив ағымдағы `expectedRevision` мәнін талап етеді
және ревизияны бірге арттырады. `presetId` көрсетілген run оны worker ішінде
орындалу сәтінде job иесімен шешеді; жоқ немесе архивтелген пресет ескірген
анықтаманы іске қосудың орнына job-ты типтелген қатемен аяқтайды.

## API

Барлық жолдар database аутентификациясын, rate limiting-ті және
`Cache-Control: no-store` талап етеді. Run-дарға әдейі бөлек endpoint жоқ:
олар бар compute-jobs беті арқылы өтеді.

| Әдіс | Жол | Мақсаты |
| --- | --- | --- |
| `GET` | `/api/screener/presets?limit=100` | Пресеттер тізімі, архивтелмегендер алдымен |
| `POST` | `/api/screener/presets` | `clientId` бойынша идемпотентті жасау |
| `PUT` | `/api/screener/presets/:id` | `expectedRevision` арқылы анықтаманы ауыстыру |
| `POST` | `/api/screener/presets/:id/archive` | `expectedRevision` арқылы архивтеу |
| `POST` | `/api/jobs` | Run кезекке қою: `{kind: "screener", clientRequestId, request}` |
| `GET` | `/api/jobs/:id` | Run polling; нәтижесі — `screener-run-result-v1` |
| `POST` | `/api/jobs/:id/cancel` | Best-effort бас тарту |

Скринер сұрауының денесі 32 768 байтпен, сақталатын анықтама 16 384 байтпен,
пресеттер тізімінің беті 100 жазбамен шектелген. Белгісіз өрістер мен
query параметрлері қабылданбайды.

## Квоталар және бета шектеулер

| Шекара | Шектеу |
| --- | ---: |
| Бір иенің белсенді пресеттері | 40 |
| Глобалды белсенді пресеттер | 400 |
| Бір run-дағы әлем символдары | 200 |
| Бір анықтамадағы фильтрлер | 12 |
| Нәтиже жолдары | 100 |
| Қатар candle оқулары | 6 |
| Run деректер бюджеті | 90 с |
| Run wall-time (worker) | 120 с |
| Бір символға жабылған candle | warm-up + 3, 50–600 аралығында |

Пресеттің глобалды белсенді күйіне кіру алерттерден бөлек namespace-і бар
арнайы PostgreSQL advisory lock арқылы сериализацияланады. Бұл — консервативті
бета шектеулер; оларды көтермес бұрын R11 құжатталған 100 пайдаланушы
жүктемесінен өтуге тиіс.

## Run тарихы және retention

Run тарихы бөлек ішкі жүйе емес: бэктесттерді басқаратын сол шектелген
compute-job retention. Job-тың толық артефакттары 30 күн сақталады, бір иеге
ең көбі 200 толық артефакт және 256 MiB. Тазартылған run-дар нәтиже болмаған
сияқты көрінудің орнына HTTP `410` жауабын беретін auditable tombstone
қалдырады; tombstone-дар 90 күн, бір иеге ең көбі 1 000 сақталады.

## Графикпен паритет

Нәтиже жолын басу графикті сол символмен, сол таймфреймбен және анықтама
фильтрлерінен шығарылған индикатор конфигурацияларымен (RSI, EMA/SMA
overlay-лары, MACD, ATR) ашады. Сервер мен график индикатор мәндері golden
паритет фикстурасымен байланған: бекітілген 300 candle сериясында скринер
қолданатын `@saltanatbotv2/strategy-core` RSI/EMA/SMA/MACD/ATR нәтижелері
браузер графигінің индикатор математикасымен екеуі де анықталған жерде тең
болуға тиіс. Сондықтан скринер сәйкестігі график жабылған candle-дарда
көрсететінге сай келеді; графиктің әлі қалыптасып жатқан candle-ы скринингке
кірмейді.

## PostgreSQL schema 14

Schema 14 бір `screener_presets` кестесін қосады: пресет ID мен client ID
бойынша owner-scoped бірегейлік, өлшемі тексерілетін `jsonb` анықтама,
анықтама хэші, оң ревизиялар және архив уақыт белгісі. Миграция аддитивті;
schema 13 файлдары өзгертілмеген. Upgrade және rollback schema 13 үшін
құжатталған backup-first және restore-into-replacement тәртібіне сүйенеді —
rollback үшін schema 14 жолдарын ешқашан өшірмеңіз және `schema_migrations`
мәнін азайтпаңыз.

Қараңыз: [MIGRATIONS.md](../MIGRATIONS.md),
[BACKUP_RESTORE.md](BACKUP_RESTORE.md) және [RELEASING.md](../RELEASING.md).

## Верификация

Release gate мыналарды қамтиды:

- скринер схемаларының қатаң контракт генерация тексерістері;
- exact-key API схема шектері мен белгісіз өрістерді қабылдамау тесттері;
- әр фильтр тесттері, соның ішінде warm-up қолжетімсіздігі, екі жабылған
  candle cross семантикасы, детерминистік сұрыптау/қысқарту және
  strategy-core/график golden паритет фикстурасы;
- маршрут тесттері: owner guard, қате кестесі, `no-store`, проекция
  ағып кетуі, 413 дене шектеуі;
- нақты unprivileged PostgreSQL schema 14 миграциясы, екі иенің оқшаулануы,
  квота, идемпотенттілік, ревизия қақтығысы және архив тесттері;
- worker executor тесттері: нәтиже пішіні, бюджет таусылуы, пресет шешу
  қатесі;
- браузер клиенті, workspace және EN/RU/KK локализация тесттері, сондай-ақ
  қолжетімділік тексерістері бар толық скрин→график сценарийі.

R5.2.1 қабылданды және қорғалған `r5b-schema14-20be5b1` slot-ынан
PostgreSQL schema 14 нұсқасында деплой жасалды
(коммит `20be5b1d2fb87df38cc298953dfe7a2f414dd831`, exact-SHA CI run
`29584556266`, 6/6 job). Қабылдау мен cutover жазбасы —
[R5.2.1 evidence](../evidence/R5_2_1_TECHNICAL_SCREENER.md).

Сканды алертке айналдыру енді R5.3a арқылы жеткізіледі: «Осы сканнан алерт
жасау» әрекеті ағымдағы жарамды анықтаманы kind `screener` тұрақты серверлік
ережесіне айналдырады; ол сканды таймфрейм каденциясымен қайта орындайды және
сәйкес символдар жиыны өзгергенде алерт event жасайды; жеткізу R5.3b-ге дейін
тек in-app. Семантика, белгісіз символдарды алып жүру, қолжетімділік шегі,
cooldown және бір owner-ға 5 / глобалды 40 квоталары
[Скринер алерттерінде (R5.3a)](./ALERTS.md) құжатталған. R5.3a қабылданды
және миграциясыз деплой жасалды: production өзгеріссіз PostgreSQL schema 14
нұсқасында қорғалған `r5c-schema14-86712ba` slot-ынан жұмыс істейді (коммит
