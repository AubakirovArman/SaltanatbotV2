# Рабочая область арбитражных исследований

Раздел **Скринер** работает без API-ключей и не отправляет ордера. Basis-режим проверяет общие
USDT-пары Binance и Bybit и рассчитывает четыре направления: spot → perpetual внутри Binance,
между Binance/Bybit в обе стороны и внутри Bybit. В интерфейсе также есть направленная симуляция
трёхногих внутрибиржевых циклов по REST top-book, книги нативных спредов Bybit и лаборатория
паритета европейских опционов. Все значения capacity/borrow/margin/rates/fees в опционной форме —
допущения пользователя, а не live account evidence.

Browser-режимы basis, triangular и native spreads используют общее локальное исследовательское
пространство: пользовательские столбцы, компактную тепловую карту, граф последовательности ног и
сравнение текущих кандидатов. Режимы options, funding и continuous имеют отдельные панели и пока не
используют эти пресеты и визуализации. До двенадцати именованных пресетов общего пространства
сохраняют только ограниченные фильтры и настройки представления в
версионированной локальной схеме; данные версии 1 мигрируют, а повреждённые или слишком большие
значения безопасно сбрасываются. API-ключи в пресеты не попадают, право исполнения они не дают.
Тепловая карта показывает точные значения и место в рейтинге, поэтому не полагается только на цвет;
у графа есть текстовая и табличная альтернатива. В скрытой вкладке частые обновления визуализации
приостанавливаются. В basis-режиме дополнительно виден агрегированный read-only lifecycle, который
сохраняет `executionPermission: false` и не является кнопкой отправки ордера.

На мобильном basis-результаты по умолчанию показаны компактными карточками, поэтому важные поля и
действия строки не растягивают всю страницу. Переключатель **Полная таблица** открывает полноценную
семантическую таблицу внутри собственного горизонтального scroll-контейнера; возврат к
**Карточкам** удаляет таблицу из активного DOM, а не оставляет два скрытых набора результатов.

Сворачиваемый справочник **Типы вилок** рядом с переключателем режимов связывает разговорные
термины с реальными движками: двойная вилка — двухногий pairwise route, тройная — три конверсии
triangular cycle, внутрибиржевая описывает место исполнения, а не отдельную стратегию, multi-leg —
ограниченный исследовательский маршрут из четырёх–восьми ног. Для каждого типа показаны риски
глубины, комиссий, неатомарности, капитала и восстановления. Канонические определения и разделение
текущего/планируемого приведены в [классификации арбитража](ARBITRAGE_TAXONOMY.md).

Треугольный REST-результат — только исследовательский кандидат: контракт явно содержит
`rest-top-book`, `rest-snapshot`, `sequenceVerified: false` и
`executionStatus: non-executable-candidate`. Он не выдаётся за sequence-verified исполнение.

Для собственных исследовательских клиентов доступны детерминированный pairwise evaluator, общий
credential-free public API всех адаптеров из [матрицы возможностей](../VENUE_CAPABILITIES.md) и
TypeScript SDK в `packages/arbitrage-sdk`. Режим **Live-маршруты** использует только отдельное
подмножество continuous-протоколов, зафиксированное в
[машиночитаемом реестре](../CAPABILITY_TRUTHS.json), и показывает выбранные оператором публичные
потоки, состояния источников и совместимые типы маршрутов; это отдельный исследовательский экран,
а не автоматические строки исполняемого basis-скринера.

Скринер показывает грязный спред, издержки по отдельным профилям комиссий, чистый спред, доступный объём
в лучших ценах и funding. Объём ограничен меньшей из двух сторон.
Разница больше 20% по модулю отбрасывается: это чаще коллизия одинаковых тикеров разных активов,
реденоминация или устаревший рынок, а не исполнимая возможность.
Строки являются кандидатами для исследования, а не обещанием исполнения. Кандидат без двух venue
timestamp остаётся видимым как `unverified`, ранжируется ниже `fresh`, не запускает алерт и не
проходит paper-gate.
Для межбиржевого сопоставления также нужна проверенная экономическая идентичность актива. Начальный
allowlist включает только BTC и ETH; остальные одинаковые тикеры блокируются, пока не проверены их
происхождение, реденоминация и условия расчёта. Внутрибиржевые маршруты этим allowlist не
ограничиваются: обе ноги должны иметь одну строгую venue-native identity и совместимые quote,
settlement, direction, multiplier и единицы количества в одном снимке реестра.

```text
грязный спред (bp) = (bid фьючерса - ask спота) / ask спота × 10 000
чистый спред (bp)  = грязный спред - оценка общих издержек
согласованный base quantity = min(размер ask спота, размер bid фьючерса)
доступный объём USD         = согласованный base quantity × ask спота
```

В браузере отдельно задаются taker-комиссии спота и фьючерсов Binance/Bybit, запас на
проскальзывание туда-обратно, срок удержания, годовая ставка финансирования/займа и фиксированная
стоимость перевода. Вход и выход учитываются автоматически. Funding засчитывается только за
дискретные расчёты внутри выбранного горизонта, если `nextFundingTime` и интервал подтверждены
реестром. Непроверенная положительная ставка не получает предполагаемую прибыль, а непроверенная
отрицательная ставка при ненулевом горизонте списывается минимум за один settlement даже без
`nextFundingTime`. Будущая ставка всё равно
неизвестна, поэтому это оценка, а не гарантия.

По умолчанию сервер сортирует по ожидаемой исполнимой прибыли в долларах: чистый edge умножается
на меньший из выбранного notional и top-book capacity. Фильтры применяются до `limit`, а ответ
явно содержит `totalOpportunities` и `truncated`, поэтому обрезанный список нельзя принять за весь
рынок. UI отдельно показывает cost waterfall, оценку spot-капитала, начальной маржи и safety buffer,
а также сценарии 100/75/50/25/0% схождения basis. Это пользовательская модель, не данные счёта.

REST параллельно создаёт начальный снимок, после чего сервер держит по одному общему публичному
WebSocket для каждого рынка Binance/Bybit. У Binance Spot REST `bookTicker` нет venue timestamp,
поэтому bootstrap-кандидат остаётся `unverified`; Spot `ticker` WebSocket даёт event time и текущие
bid/ask и может повысить его до `fresh`. Для Binance Futures используется `bookTicker`, для Bybit — V5 ticker.
Сервер
отправляет heartbeat, признаёт поток здоровым только после валидных данных, закрывает молчащий
канал, восстанавливает соединение с jittered exponential backoff и группирует частые тики
перед `/arbitrage-stream`. REST обновляется раз в 30 секунд для обнаружения новых пар и как
страховочная граница вместо прежнего двухсекундного polling. Временный полный
отказ может показать последний успешный снимок не старше 30 секунд с явной пометкой. В скрытой
вкладке браузер приостанавливает поток. Для каждой ноги отдельно указано, получен ли exchange
timestamp от биржи. Если его нет, поле не синтезируется из локального времени, а строка становится
`unverified`. Возраст и перекос берут худшее значение по venue timestamp и неизменяемому
`receivedAt`.

Экран basis также раз в 30 секунд запрашивает `GET /api/arbitrage/clock-health`, пока вкладка
видима. Для Binance, Bybit, OKX, Deribit, Kraken и Coinbase, у которых есть официальные публичные endpoint времени,
отдельно показаны состояния
`calibrated/degraded/expired/unavailable`, измеренный RTT и консервативная неопределённость
смещения часов. В скрытой вкладке опрос прекращается. Публичная диагностика ограничена по времени
и размеру и никогда не выдаёт локальное время за биржевое. REST-ранжирование basis, обновления
WebSocket и серверные алерты пока используют Binance/Bybit-калибровку и консервативные
скорректированные интервалы age/skew.
При отсутствующей или деградировавшей калибровке строка fail-closed становится `unverified`; все
возможности по-прежнему остаются только исследовательскими.

Сервер также ведёт ограниченный lifecycle: `first-seen`, `confirmed`, `decaying`, `expired`.
Подтверждение требует разных наблюдений и полной картины market data плюс instrument identity.
`GET /api/arbitrage/lifecycle` доступен только для чтения и всегда возвращает
`executionPermission: false`.

## Глубина, алерты и paper-позиции

Для выбранной трёхногой вилки `POST /api/arbitrage/triangular/verify-depth` запускает второй этап:
восстанавливает три L2-стакана из snapshot и WebSocket delta, проверяет sequence и текущую generation,
затем повторно считает глубину, комиссии, шаги/минимумы, свежесть и рассинхронизацию. Интерфейс
показывает sequence/generation каждой ноги и явную причину, если top-book кандидат исчез. Даже
прошедший маршрут остаётся `readOnly`, `researchOnly`, `executable: false`: ключи, балансы и ордера
не используются, а три последовательных ордера не становятся атомарными.

`GET /api/arbitrage/depth` по явному запросу получает ограниченную глубину каждого стакана через
on-demand sequence-reconstructed L2, выводит одну согласованную базовую величину, округление по общему шагу, остаточную дельту, VWAP, худшую цену,
число уровней, исходное время каждой книги, age/skew и проскальзывание. Вход проходит spot asks и
perpetual bids; закрытие для точного открытого quantity проходит spot bids и perpetual asks. При
устаревшей/перекошенной/неполной глубине или несовпадении объёма ног paper-действие блокируется.
Без venue timestamp либо подтверждённой непрерывности sequence у обеих книг качество остаётся
`unverified`, `complete: false`, поэтому paper-вход и выход блокируются. Binance Spot связывает
REST snapshot с diff-depth по `lastUpdateId + 1`, USD-M затем проверяет `pu`, а Bybit Spot/Linear
сбрасывает книгу по V5 WebSocket snapshot и принимает только непрерывный `u`. Gap или reconnect
немедленно отзывает предыдущую книгу. REST fallback явно помечается `rest-snapshot`,
`sequenceVerified: false`, остаётся display-only и не может открыть/закрыть paper-позицию.
До запроса стаканов endpoint требует актуальные записи реестра для обоих точных инструментов,
подтверждённые lot step и биржевые минимумы; при отсутствии или непроверенной metadata запрос
завершается fail-closed. `quantityStepSource` сохраняется как provenance контракта, но успешный
публичный анализ имеет источник `instrument` и `precisionVerified: true`.
Каждый ответ также содержит стабильные ID обоих инструментов, native/reviewed `identityScope` и
доступный проверенный economic asset ID. Перед открытием или закрытием paper-позиции браузер
сверяет их вместе с symbol, биржами, типами рынков, направлением и сторонами ног с исходным
запросом. Отключение клиента отменяет upstream-работу после ухода последнего общего подписчика, а
лишние уникальные запросы стаканов получают явный overload-ответ вместо неограниченной очереди.

Алерт срабатывает только при пересечении настроенного чистого порога. Уведомление рабочего стола
требует разрешения браузера. Удалённая доставка доступна только в авторизованной paper-trade
сессии. До 50 постоянных правил работают и при закрытой вкладке; состояние пересечения хранится
отдельно для каждой пары «правило + маршрут». Durable outbox сначала сохраняет событие, затем
повторяет неудачную доставку после перезапуска и показывает статусы
queued/sending/retrying/delivered/failed/cancelled. Возможен дубликат, если внешний канал принял
сообщение, а процесс упал до фиксации `delivered`: это граница at-least-once. Алерты не отправляют
ордера.

`GET /api/arbitrage/history` читает ограниченную семидневную историю из SQLite. При активном потоке
50 лучших маршрутов записываются не чаще раза в минуту, старые данные удаляются, а панель глубины
показывает последние 24 часа выбранного маршрута.

Исторический basis replay — отдельный детерминированный backend-модуль. Неизменяемый manifest
schema v4 связывает digest событий, файлов, снимка реестра, версий адаптеров, canonical economic
asset ID и версионированные изменения шага/минимального количества/минимального номинала.
Listing, изменения ограничений и delisting применяются point-in-time, вход и выход проходят
записанную глубину, а PnL учитывает только подтверждённый funding, чей settlement `exchangeTs`
попал в полуоткрытый интервал `[openedAt, actualClosedAt)`. Запоздавший record сохраняет
`receivedAt`, но не переносит расчёт и не меняет
уже выбранные вход/выход.

Paper-позиции хранятся локально как ограниченный, версионированный append-only журнал событий.
Вход и выход используют согласованный VWAP двух стаканов. Funding добавляется только вручную как
подтверждённое событие с временем расчёта, ставкой и reference price; текущая ticker-ставка не
выдаётся за полученный платёж. Восстановление повторно сворачивает журнал в тот же PnL и отклоняет
дублированные или изменённые события. Панель показывает реализованный и открытый PnL, win rate и средний
результат закрытых позиций. Это не биржевой счёт.
Live-таблица выводит по 50 строк на страницу, чтобы частые обновления не перестраивали сотни DOM-строк.

## Исследование многошаговых циклов

`POST /api/arbitrage/n-leg/evaluate` принимает точные spot-метаданные и полные стаканы с
подтверждённой последовательностью, после чего ограниченно ищет и моделирует простые циклы из
четырёх–восьми ног. Проверяются точные `(биржа, актив, единица)`, видимая глубина, шаги и минимумы,
комиссия каждой стороны и остаточная пыль. HTTP-ответ и публичный SDK всегда доступны только для
чтения и не исполняют сделки.

Изолированный серверный paper-модуль умеет преобразовать уже рассчитанный N-leg или двухногий
route-family результат в короткоживущий `paper-multi-leg-plan-v1`. Он записывает явные
детерминированные коэффициенты тестовых fill, останавливается на первой неполной ноге, журналирует
решение о компенсации и моделирует обратные fills в обратном порядке. Результат — completed,
compensated, aborted без экспозиции либо manual review с точным некомпенсированным paper-остатком.
Append-only SQLite-журнал проверяет hash и последовательность событий, имеет жёсткие лимиты и
восстанавливает незавершённые прогоны после перезапуска. Он подключён только под защищённым
`/api/trade/paper-multi-leg`: нужна роль `paper-trade`, а mutation дополнительно требует CSRF.
В разделе **Торговля** доступен строгий интерфейс на русском, английском и казахском для импорта
точного JSON-плана, просмотра recovery, списка прогонов и append-only событий. Этот журнал
принимает только точный JSON-план исследовательского движка. Находящийся в работе слайс R8 (НЕ
принят) переиспользует чистый движок, builders и валидацию этого модуля дословно для owner-scoped
durable multi-leg интентов внутри версионированного trading store — с общим резервированием
капитала, комбинированным research PnL «обе ноги + все издержки» и явной остаточной экспозицией;
карточка исследования с `paperPlan: ready` получает подтверждаемое действие «Запустить paper
multi-leg», а сервер заново строит и валидирует план fail-closed в момент apply
([ru/TRADING.md](./TRADING.md), [Канонические paper-портфели](./PAPER_PORTFOLIOS.md)). Сам
изолированный журнал, его хранилище и admin-доступ остаются байт-идентичными. Модуль не
содержит private exchange client, не принимает credentials, не входит в public SDK и не является
реальным исполнителем цепочки ордеров или live-сканером всей биржи.

## Сценарии funding-кривой

Вкладка **Funding-сценарии** позволяет выбрать до четырёх свежих perpetual-инструментов из
серверного `GET /api/arbitrage/funding-curve/universe`. Endpoint пересекает подтверждённый реестр
с адаптерами, реально реализованными Funding Curve, поэтому общая capability Binance/Bybit больше
не делает неподдерживаемый инструмент доступным. `POST /api/arbitrage/funding-curve` строит
дискретные расчёты на ограниченный горизонт и применяет
базовый, положительный и отрицательный additive stress к каждому settlement. Строгий public SDK
повторно проверяет interval, freshness, единицы, timestamps, арифметику сценария и постоянный
non-executable envelope.

Разница ставок сравнивается только при одной точной reviewed `economicAssetId`. Положительный
funding означает, что long платит short, поэтому меньшая накопленная ставка подписывается как
исследовательский long, а большая — как short. Это не сделка и не P&L: basis входа/выхода,
комиссии, margin, liquidation, capital, borrow и fill risk в число не входят. Continuous или
inferred расписание отклоняется fail-closed вместо выдуманного интервала.

## Непрерывные межбиржевые маршруты

Вкладка **Live-маршруты** опрашивает `GET /api/arbitrage/route-families/live` только пока страница
видима. Ограниченный allowlist инструментов и комиссия задаются оператором на сервере; браузер не
может их изменить. Economic identity берётся только из центрального точного versioned-каталога:
строка environment обязана совпасть с ним и не может объявить новое равенство активов. Без allowlist WebSocket-подписки не
запускаются. Ошибка реестра, истёкшая проверка identity, gap или неподдерживаемый инструмент удаляют
данные безопасно, без повторного использования устаревшего маршрута.

Стаканы OKX, Gate.io и Deribit считаются sequence-ready только после проверки непрерывности их
протокола. Полные block snapshot Hyperliquid видны как исследовательские данные, но не выдаются за
sequence proof. Если оба публичных стакана свежи по локальному времени получения, укладываются в
лимит рассинхронизации ног, имеют подтверждение sequence или checksum и относятся к текущей
generation соединения, сервер добавляет оценку `market-only`. Он согласует нормализованные модели
количества и берёт максимальный общий base quantity, видимый на текущих ask покупки и bid продажи.
Результат показывает разницу между quote value продажи по short bid и покупки по long ask, а также
соответствующий basis входа до и после публичной quote-equivalent оценки taker-комиссий из operator
environment. Это разницы стоимости входа, а не торговый финансовый результат или ожидаемая
прибыль. Расчёт охватывает только top-book и вход, а не полную глубину или полный цикл. Актив
комиссии не подтверждён, влияние комиссии на base/quote exposure не включено; профиль также не
подтверждает account tier, скидку или rebate.

При жёстком лимите 24 инструментов сервер перечисляет весь совместимый universe (не более 552
упорядоченных пар), оценивает каждый маршрут и только затем публикует лучший ограниченный набор.
Сначала сравнивается fee-adjusted разница entry quote value, затем basis, видимая capacity,
качество continuity и freshness. Отдельные счётчики evaluated/published честно показывают
усечение: порядок family или route ID больше не может отбросить лучший рынок до расчёта экономики.

Каждая строка `market-only` содержит две записи economic identity в строгом порядке long/short.
Запись связывает `instrumentId`, `economicAssetId`, reviewed status, source, version, `asOf` и
`validUntil` и должна быть действительна на `evaluatedAt`. Невалидная, ещё не начавшая действовать
или истёкшая provenance блокируется fail-closed. Производные capacity, quote value, оценочные
комиссии и basis должны оставаться конечными, положительными там, где это требуется, и
арифметически согласованными. Overflow, underflow или подделанное производное поле отклоняются, а
не публикуются. Строгий SDK отдельно сверяет упорядоченную provenance и повторно считает
арифметику. Ошибка continuity, generation, freshness, quantity или биржевого минимума также даёт
явный market-data blocker вместо результата по стоимости входа.

`market-only` не означает, что маршрут исполним. Каждая оценка сохраняет `readOnly: true`,
`researchOnly: true`, `executable: false` и `strategyStatus: blocked`; ордера на этой границе не
поддерживаются. Расчёт не знает подтверждённые балансы, капитал, inventory, сеть/вывод, займ, margin
дериватива, funding на полном горизонте, схождение, expiry/delivery, стоимость выхода, фактические
fills и не гарантирует прибыль. Эти отсутствующие strategy evidence остаются видимыми blockers даже
при полной рыночной паре.

Верхний runtime и discovery snapshot публикуют одну и ту же границу coverage: `complete`, `current`,
`retainedPriorDiscovery` и ограниченную причину (`complete`, отключённая/невалидная конфигурация,
ожидание/ошибка refresh или частичный набор инструментов). Успешный частичный refresh актуален, но
неполон. Ошибка следующего registry refresh может сохранить предыдущее discovery для наблюдения,
однако помечает его неполным, неактуальным и явно сохранённым: это stale evidence, а не успешное
обновление. Ошибка самого первого refresh ничего не сохраняет и не выдумывает `refreshedAt`.

Lifecycle непрерывных маршрутов может использовать basis входа после оценочных комиссий как
исследовательское наблюдение, но evidence каждой ноги остаётся неполной, а `actionable: false`.
Кандидат, заблокированный market data и не имеющий полезной evidence, пропускается вместо
синтетического маршрута с нулевой оценкой, поэтому он не ломает отдельное корректное наблюдение.
Его реальные blocker codes всё равно попадают в failure coverage lifecycle. Причины runtime
refresh, non-live источники, исключённые/отклонённые входы, truncation кандидатов/расчётов и stale
market codes переходят в incomplete/stale/truncated coverage. Поэтому строки и lifecycle описывают
ограниченные публичные рыночные evidence, а не торговый сигнал или разрешение на ордер.

Отдельный ежедневный/manual credential-free canary наблюдает по одной выбранной цели на каждой из
девяти generic continuous-бирж. Для Spot обязателен публичный стакан, для дериватива — стакан и
публичное funding-наблюдение; явно reviewed dYdX остаётся book-only research-target. Сохраняемый
schema-v3 JSON-артефакт фиксирует требования, environment, integrity, continuity и постоянные поля
`credentialsUsed: false`, `executionAttempted: false`, `soakClaimed: false`,
`mainnetReadinessClaimed: false`. Локальный прогон 2026-07-14 прошёл для OKX, Gate, Hyperliquid,
Deribit public testnet, Coinbase, dYdX, KuCoin и MEXC. Kraken остался недоступен через TLS egress
этого сервера. Live-прогоны выявили и получили regression-тесты для binary-marked JSON KuCoin,
connection-global sequence Coinbase и гонки snapshot/delta bootstrap MEXC. Один прогон не является
soak или execution-readiness evidence.

Скринер работает только для исследования и не отправляет ордера. Котировки двух бирж не атомарны.
Комиссии, реальная глубина, funding, доступность займа, переводы, задержка и риск ликвидации могут
полностью убрать найденную разницу. Положительный результат не является гарантией прибыли.
