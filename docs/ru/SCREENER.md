# Технический скринер по запросу

English reference: [SCREENER.md](../SCREENER.md).
Қазақша нұсқа: [kk/SCREENER.md](../kk/SCREENER.md).

Документ описывает технический скринер R5.2.1, добавленный PostgreSQL
schema 14. Это исключительно исследовательский индикаторный скан по спотовой
USDT-вселенной Binance, запускаемый по запросу. Каждый payload несёт
`researchOnly: true` и `executionPermission: false`. Скринер не может выставить
ордер, занять актив, изменить маржу, подписать запрос биржи или выдать торговую
роль.

## Что входит в R5.2.1

Поддерживается одна форма скана:

- kind `technical` по публичным спотовым рынкам Binance, только last price;
- таймфреймы `5m`, `15m`, `1h`, `4h` и `1d`;
- вселенная из 10–200 USDT-символов каталога, отсортированных по возрастанию;
- 1–12 фильтров семи видов, оцениваемых только по закрытым свечам;
- детерминированная сортировка, не более 100 строк результата;
- серверные пресеты владельца с ревизиями и архивом;
- ограниченная история запусков через существующее retention compute-jobs;
- переход в график с сохранением символа, таймфрейма и индикаторов.

Не входит в R5.2.1: запланированные и постоянно повторяемые сканы,
Bybit-вселенная и не-USDT котировки. Превращение скана в серверный алерт сюда
тоже не входило; оно поставляется отдельно инкрементом R5.3a как rule kind
`screener` — см. [Скринер-алерты (R5.3a)](./ALERTS.md). Контракт определения в этом
инкременте принимает только `exchange: "binance"`, `marketType: "spot"` и
`priceType: "last"`; схема оставляет место для новых бирж без слома версии.

Скринер не читает ключи биржи. Он использует один публичный batch 24-часовых
тикеров и строгие публичные закрытые свечи и отклоняет синтетические,
кэшированные, приватные или подменённые данные.

## Граница HTTP без HTTPS

Текущий pre-HTTPS релиз намеренно не настраивает TLS. Пароль и session cookie
должны передаваться только внутри доверенной локальной сети, приватного VPN или
SSH-туннеля. Нельзя выставлять эту сборку как общедоступный login-сервис в
Интернет и нельзя добавлять в неё приватные ключи биржи. HTTPS — отдельный
release gate.

## Поток данных

```text
авторизованный браузер
  -> POST /api/jobs (kind screener, идемпотентный clientRequestId)
  -> долговечный compute job владельца
  -> lease исследовательского worker
  -> один публичный 24h ticker batch + строгие закрытые свечи по символам
  -> чистая детерминированная оценка фильтров
  -> ограниченный сериализованный screener-run-result-v1
  -> polling GET /api/jobs/:id
  -> таблица результатов / переход в график
```

Запуск — это обычный compute job. Браузер ставит его в очередь с идемпотентным
`clientRequestId` (8–128 символов), опрашивает job каждые 2 секунды с клиентским
дедлайном 120 секунд и запрашивает best-effort cancel, если владелец бросил
запуск. Одинаковые ожидающие payload дедуплицируются подсистемой jobs; у одного
владельца не более пяти активных и одного выполняющегося job.

## Справочник определения

Документ `screener-definition-v1` разбирается по точным ключам; неизвестные поля
отклоняются. Он содержит:

| Поле | Значение |
| --- | --- |
| `name` | 1–120 символов |
| `exchange`, `marketType`, `priceType` | только `binance`, `spot`, `last` |
| `timeframe` | `5m`, `15m`, `1h`, `4h` или `1d` |
| `universeLimit` | целое 10–200 |
| `sort` | ключ `quoteVolume24h`, `change24hPercent`, `lastClose`, `symbol`, `rsi` или `atrPercent`; направление `asc`/`desc` |
| `filters` | 1–12 объектов фильтров |
| `researchOnly`, `executionPermission` | обязаны быть `true` / `false` |

Числовые пороги — десятичные строки того же формата, что и в контрактах
алертов; они никогда не округляются молча.

## Справочник фильтров

| Kind | Параметры | Совпадение |
| --- | --- | --- |
| `price` | `min?`, `max?` (минимум одна граница) | последний close внутри границ |
| `quote-volume-24h` | `min` (USDT) | 24h quote volume ≥ `min` |
| `change-24h-percent` | `min?`, `max?` (−100…10000, минимум одна) | 24h изменение внутри границ |
| `rsi` | `period` 2–200, `condition` `above`/`below`, `value` 0–100 | RSI по close удовлетворяет условию |
| `ma-cross` | `fastType`/`slowType` `ema`/`sma`, `fastPeriod` < `slowPeriod`, `state` | `fast-above`, `fast-below`, `crossed-up` или `crossed-down` |
| `macd` | `fast` < `slow`, `signal` | `histogram-above-zero`, `histogram-below-zero`, `crossed-up`, `crossed-down` |
| `atr-percent` | `period` 2–200, `condition`, `value` 0–1000 | ATR в процентах от последнего close удовлетворяет условию |

Последняя закрытая свеча — свеча оценки. Cross-состояния сравнивают ровно две
последние закрытые свечи: `crossed-up` означает, что fast не был выше на
предыдущей закрытой свече и выше на свече оценки. Индикаторная математика
берётся из `@saltanatbotv2/strategy-core` (RSI, SMA, EMA, MACD, ATR) — того же
семейства пакетов, что и график; именно это делает паритет с графиком
доказуемым.

## Закрытые свечи и fail-closed недоступность

Скринер оценивает только финальные свечи. Формирующийся хвост отбрасывается, а
оставшееся окно проходит ту же валидацию закрытых свечей, что и evaluator
алертов: отсутствующие, формирующиеся, будущие, устаревшие, разрывные,
превышающие размер или искажённые окна дают fail closed.

Символ с неполными данными становится **недоступным, а не нулевым**. Индикатор
внутри окна прогрева, отсутствующая строка тикера или исчерпанный бюджет
запуска исключают символ из оценки с явной причиной вместо подстановки
фиктивного `0` в фильтр. Результат содержит счётчики
`universe.requested/evaluated/matched/unavailable` и карту `unavailableReasons`
с точными причинами, в том числе:

| Причина | Значение |
| --- | --- |
| `ticker-unavailable` | нет валидной публичной строки 24h тикера |
| `missing-candles` | провайдер не вернул закрытые свечи |
| `indicator-warm-up` | не хватает закрытых свечей для индикатора |
| `run-budget-exhausted` | 90-секундный бюджет закончился до этого символа |
| `upstream-unavailable` | ограниченное публичное чтение не удалось |
| `row-out-of-range` | совпавшее значение не представимо в ограниченном контракте |

Если сам batch 24-часовых тикеров не разбирается, весь запуск падает fail
closed, а не продолжается на частичных ценовых данных.

## Детерминизм и форма результата

Результат — документ `screener-run-result-v1` с хэшем определения, временем
генерации, диапазоном закрытых свечей и не более чем 100 строками. Строки
сортируются по ключу из определения, undefined-значения в конце, при равенстве
— символ по возрастанию, поэтому одинаковая вселенная с одинаковыми свечами
даёт одинаковый результат. Если совпало больше 100 символов, результат ставит
`rowsTruncated: true` вместо молчаливого отбрасывания данных. Десятичные
метрики форматируются `toFixed(8)` с обрезкой хвостовых нулей.

## Владение и авторизация

Каждое чтение и запись пресета выводит владельца из аутентифицированной
database-сессии и дополнительно требует заголовок `X-SBV2-Expected-User` с тем
же user ID. Мутации требуют обычный CSRF-заголовок. Транзакции репозитория
повторно проверяют активность пользователя, `must_change_password = false`,
текущую authorization revision и совпадение актора с владельцем — точно как
репозиторий алертов. У администраторов нет пути чтения или мутации чужих
пресетов. Публичные проекции никогда не содержат `owner_user_id`,
`authorization_revision` и `definition_hash`.

Создание пресета идемпотентно по `clientId`: повтор того же определения
возвращает существующий пресет, а другое определение под тем же `clientId` —
конфликт `409`. Обновление и архив требуют текущую `expectedRevision` и
увеличивают ревизию на один. Запуск по `presetId` разрешает его в worker в
момент выполнения от имени владельца job; отсутствующий или архивированный
пресет завершает job типизированной ошибкой вместо запуска устаревшего
определения.

## API

Все пути требуют database-аутентификацию, rate limiting и `Cache-Control:
no-store`. У запусков намеренно нет отдельного endpoint: они идут через
существующую поверхность compute-jobs.

| Метод | Путь | Назначение |
| --- | --- | --- |
| `GET` | `/api/screener/presets?limit=100` | Список пресетов, неархивные первыми |
| `POST` | `/api/screener/presets` | Идемпотентное создание по `clientId` |
| `PUT` | `/api/screener/presets/:id` | Замена определения по `expectedRevision` |
| `POST` | `/api/screener/presets/:id/archive` | Архив по `expectedRevision` |
| `POST` | `/api/jobs` | Постановка запуска: `{kind: "screener", clientRequestId, request}` |
| `GET` | `/api/jobs/:id` | Polling запуска; результат — `screener-run-result-v1` |
| `POST` | `/api/jobs/:id/cancel` | Best-effort отмена запуска |

Тело запроса скринера ограничено 32 768 байтами, хранимое определение —
16 384 байтами, страница списка пресетов — 100 записями. Неизвестные поля и
query-параметры отклоняются.

## Квоты и бета-лимиты

| Граница | Лимит |
| --- | ---: |
| Активные пресеты на владельца | 40 |
| Глобально активные пресеты | 400 |
| Символов вселенной на запуск | 200 |
| Фильтров в определении | 12 |
| Строк результата | 100 |
| Параллельных чтений свечей | 6 |
| Бюджет сбора данных запуска | 90 с |
| Wall-time запуска (worker) | 120 с |
| Закрытых свечей на символ | warm-up + 3, в пределах 50–600 |

Вход в глобально активное состояние пресета сериализуется отдельным advisory
lock PostgreSQL с namespace, отличным от алертов. Это консервативные
бета-лимиты; прежде чем их поднимать, R11 обязан пройти задокументированную
нагрузку на 100 пользователей.

## История запусков и retention

История запусков — не отдельная подсистема, а то же ограниченное retention
compute-jobs, что и у бэктестов. Полные артефакты job хранятся 30 дней, не
более 200 полных артефактов и 256 MiB на владельца. Очищенные запуски
оставляют auditable tombstone, отвечающий HTTP `410`, вместо вида, что
результата не было; tombstone хранятся 90 дней, не более 1 000 на владельца.

## Паритет с графиком

Клик по строке результата открывает график с тем же символом, тем же
таймфреймом и конфигурациями индикаторов из фильтров определения (RSI,
overlays EMA/SMA, MACD, ATR). Значения индикаторов сервера и графика связаны
golden-фикстурой паритета: на фиксированной серии из 300 свечей выводы
RSI/EMA/SMA/MACD/ATR из `@saltanatbotv2/strategy-core`, которые использует
скринер, обязаны совпадать с индикаторной математикой браузерного графика
везде, где обе определены. Совпадение скринера поэтому соответствует тому, что
график показывает на закрытых свечах; ещё формирующаяся свеча графика в
скрининг не входит.

## PostgreSQL schema 14

Schema 14 добавляет одну таблицу `screener_presets` с owner-scoped
уникальностью ID пресета и client ID, `jsonb`-определением с проверкой
размера, хэшем определения, положительными ревизиями и меткой архива.
Миграция аддитивна; файлы schema 13 не тронуты. Upgrade и rollback следуют той
же дисциплине backup-first и restore-into-replacement, что задокументирована
для schema 13 — никогда не удаляйте строки schema 14 и не уменьшайте
`schema_migrations` ради отката.

См. [MIGRATIONS.md](../MIGRATIONS.md),
[BACKUP_RESTORE.md](BACKUP_RESTORE.md) и [RELEASING.md](../RELEASING.md).

## Верификация

Release gate включает:

- строгие проверки генерации контрактов скринера;
- тесты границ exact-key API-схем и отклонения неизвестных полей;
- тесты каждого фильтра, включая warm-up недоступность, cross-семантику двух
  закрытых свечей, детерминированную сортировку/усечение и golden-фикстуру
  паритета strategy-core/график;
- тесты маршрутов: owner guard, таблица ошибок, `no-store`, утечки проекций,
  лимит тела 413;
- реальные unprivileged PostgreSQL-тесты миграции schema 14, изоляции двух
  владельцев, квот, идемпотентности, конфликтов ревизий и архива;
- тесты executor в worker: форма результата, исчерпание бюджета, ошибка
  разрешения пресета;
- браузерные тесты клиента, workspace и локализации EN/RU/KK плюс сквозной
  сценарий скрин→график с проверками доступности.

R5.2.1 принят и был задеплоен на PostgreSQL schema 14 из
защищённого slot `r5b-schema14-20be5b1` (коммит
`20be5b1d2fb87df38cc298953dfe7a2f414dd831`, exact-SHA CI run `29584556266`,
6/6 jobs). Запись приёмки и cutover —
[R5.2.1 evidence](../evidence/R5_2_1_TECHNICAL_SCREENER.md).

Превращение скана в алерт теперь поставляется R5.3a: действие «Создать алерт
из этого скана» превращает текущее валидное определение в долговечное
серверное правило kind `screener`, которое повторно выполняет скан с
каденцией таймфрейма и создаёт событие алерта при изменении набора совпавших
символов; доставка — in-app, а после принятого релиза R5.3b-1 также Telegram
в привязанный чат. Семантика, перенос неизвестных символов, порог
доступности, cooldown и квоты 5 на владельца / 40 глобально описаны в
[Скринер-алертах (R5.3a)](./ALERTS.md). R5.3a принят и задеплоен без
миграции из защищённого slot `r5c-schema14-86712ba` (коммит
`86712bac3293ac8d746b638218eb66995d8e5edb`, exact-SHA CI run `29590401183`,
6/6 jobs); см. [R5.3a evidence](../evidence/R5_3A_SCREENER_ALERTS.md).
Production теперь работает на PostgreSQL schema 15 из принятого R5.3b-1
slot `r5d-schema15-cd34ec8`; см.
[R5.3b-1 evidence](../evidence/R5_3B1_TELEGRAM_DELIVERY.md).
Запланированные сканы и Bybit-вселенная остаются будущей работой.
